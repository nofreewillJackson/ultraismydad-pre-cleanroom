# Onboarding Map — how this app actually works

> **Who this is for:** a newcomer who has never seen this codebase and wants to
> understand, from scratch, *where the app starts* and *how data gets onto the
> screen* — and be able to **re-derive all of it themselves** without help.
>
> Nothing here is magic. Every claim below ends in a `file:line` you can open, or
> a command you can run. If you doubt a sentence, go check it. That habit *is* the
> skill.

---

## Start here — the toolkit and how to use it

You own two things that do this analysis for you, plus a portable playbook. No AI required.

| File | What it is |
|------|-----------|
| `scripts/trace-reachability.mjs` | a script that follows `import` lines from the entry points, lists alive-vs-dead files, and can draw the graph |
| `docs/studying-a-codebase.md` | the **general** method (any language) + per-ecosystem tools + a 10-minute "after I break it" routine |
| `ONBOARDING.md` (this file) | the answer **for this specific app** |

### The 3-command loop you'll repeat

```bash
# 1. What's alive vs dead right now?
node scripts/trace-reachability.mjs

# 2. Draw the picture (needs graphviz: sudo apt-get install graphviz)
node scripts/trace-reachability.mjs --graph && dot -Tsvg graph.dot -o graph.svg

# 3. I changed/broke a file — who depends on it?
grep -rln "theFileYouChanged" src
```

### Look at the graph

```bash
explorer.exe graph.svg      # WSL → opens in Windows browser
# or just open graph.svg / graph.png in VS Code
```
**How to read it:** boxes are files, grouped into clusters (`pages`, `components`, `lib`…);
arrows are imports. Find the box with the most arrows pointing *into* it — that's the
**hub** everything depends on. Here it's `src/lib/data.ts`. Hubs are where to start reading.

### Trace just one part instead of the whole app

```bash
node scripts/trace-reachability.mjs --roots src/pages/index.astro   # only the homepage's files
node scripts/trace-reachability.mjs --roots src/main.jsx            # only the dead React app
```

### Take this skill to other projects

The script is generic — copy it and point it at any JS/TS repo:
```bash
mkdir -p ~/bin && cp scripts/trace-reachability.mjs ~/bin/trace-deps.mjs
# in any other project:
node ~/bin/trace-deps.mjs --src src --roots src/index.ts --graph
```
For non-JS projects (Python, Go, Rust, …), `docs/studying-a-codebase.md` lists the
equivalent entry-point conventions and graph tools.

> **Housekeeping:** these files are new/untracked — `git add` the script and the two
> `.md` docs so they're safe; add `graph.dot`, `graph.svg`, `graph.png` to `.gitignore`
> (they're regenerated by the commands above).

## 0. The one big idea (read this even if you read nothing else)

Two facts explain 90% of this codebase:

1. **It's a *static site*.** All the data work happens **once, when you run
   `npm run build`** — on your machine, not in the visitor's browser. The build
   reads data, turns it into plain HTML files in `dist/`, and that's what visitors
   download. No database is contacted when someone views the site.

2. **"What references what" = follow the `import` lines.** A file *uses* another
   file when it has an `import` line pointing at it. Start at the entry points and
   follow those lines, file to file, and you've traced the whole app. Whatever you
   never reach is **dead code.**

Everything below is just those two ideas, spelled out.

---

## 1. Where does the app start? (Entry points)

There is **no single `main()` file.** This is an [Astro](https://astro.build) site,
and Astro has one rule that decides everything:

> **Every file under `src/pages/` becomes one URL.** That folder *is* the list of
> entry points. The file's path = the URL path.

So `src/pages/list.astro` → `/list`, `src/pages/log/index.astro` → `/log`, and so on.
There are **17 entry points** (see the table in §4).

### ⚠️ The decoy entry point

The file `index.html` in the project root, plus `src/main.jsx` and `src/App.jsx`,
look like an entry point — they're a complete **React app**. They are **dead.**
Astro ignores the root `index.html` and builds from `src/pages/` instead. If you
start exploring from `index.html`, you will trace an entire abandoned app and learn
nothing about the live site. (Proof: §6.)

## 2. How to reproduce this whole analysis yourself

You do **not** need an AI to do any of this. Run these and read the output.

```bash
# 0. Install deps once
npm install

# 1. GROUND TRUTH — build the site and see exactly what ships.
#    On a static site, if it's not in dist/, it isn't used.
npm run build
find dist -name '*.html'     # every page the site actually produces
find dist -name '*.js'       # the only JavaScript that runs in the browser

# 2. ALIVE vs DEAD — follow imports from the real entry points (src/pages/*).
#    This script reads each file's `import` lines and walks them recursively.
node scripts/trace-reachability.mjs
#    -> "Reachable from roots: 41/64" and a list of the 25 dead files.

# 3. PICTURE of the live import graph (needs graphviz: `sudo apt-get install graphviz`)
node scripts/trace-reachability.mjs --graph
dot -Tsvg graph.dot -o graph.svg     # open graph.svg in a browser

# 4. INSPECT THE DEAD REACT APP on its own (optional, just to confirm it's separate)
npx madge --extensions js,jsx,ts src/main.jsx     # 26 files, none of which ship
```

Two manual greps you'll use constantly:

```bash
# What does a file pull in? (read the top of any page)
grep -n "^import" src/pages/index.astro

# Who references a given file? (find a file's callers)
grep -rln "lib/data" src --include=*.astro --include=*.ts
```

> **Caveat about tools:** `madge` and `dependency-cruiser` cannot read `.astro`
> import lines, so they wrongly report `.astro` components as "orphans." The little
> `scripts/trace-reachability.mjs` in this repo exists *specifically* to handle
> `.astro`. Also, pure import-following can't see files Astro loads *by convention*
> rather than by `import` — namely `src/content.config.ts` and `src/env.d.ts`.
> Those two are **alive** even though the tracer lists them as dead.

## 3. The shape of the app: 4 layers + the data sources

Every page is built from the same four layers, top to bottom:

```
 ① ROUTE        src/pages/*.astro        one file = one URL; runs at build time
        │
        ▼
 ② SHELL        layouts/BaseLayout.astro  the <html>/<head>, fonts, global CSS
        │       components/Header.astro    nav bar + other .astro components
        ▼
 ③ DATA API     src/lib/data.ts           getPublicProjects(), getStats(), …  ← the gateway
        │       src/lib/format.ts          date / slug / markdown helpers
        │       src/lib/garden/            the digital-garden pipeline (isolated)
        ▼
 ④ DATA SOURCE  (files on disk — see below)
```

### The four data sources (where the bytes actually come from)

| # | Source on disk | Read by | Powers |
|---|----------------|---------|--------|
| A | `src/data/tracker-snapshot.json` | `src/lib/data.ts` | projects, journal/log, series, technologies |
| B | `src/content/research/*.md` | `getCollection("research")` (Astro content) | /research |
| C | `src/content/garden/*.md` | `getGardenData()` in `src/lib/garden/` | /garden |
| D | `public/videos/index.json` + `public/videos/<id>/bundle.json` | `getVideoIndex()` / `getVideoBundle()` | /video |

**Source A is the heart of the site, and it has a 3-step fallback cascade**
(`src/lib/data.ts:467`, `getPublicProjects`):

1. Read the snapshot file `src/data/tracker-snapshot.json` (`data.ts:418`). ← normal path
2. If that file is missing → query the **live Firestore** database (`data.ts:478`).
3. If Firestore fails → read a **local backup JSON** on the original author's laptop
   (`data.ts:483`; the hardcoded `/Users/ralphxu/...` path at `data.ts:158`).

Whatever it gets, it keeps only items where `visibility === "public"` (`data.ts:473`).

> The snapshot file is **not committed** to the repo — it's *generated* from Firestore
> by `npm run sync:data` (`scripts/sync-firestore-snapshot.mjs:88`). That's why the
> fallbacks exist. The full pipeline:
>
> ```
> Firestore (live DB) --[npm run sync:data]--> src/data/tracker-snapshot.json
>                                                       |
>                                              [npm run build]
>                                                       v
>                                          dist/*.html --> visitor's browser
> ```

## 4. Every route, mapped (the whole app)

`static` = one page. `dynamic` = Astro generates **one page per data item** via a
`getStaticPaths()` function (e.g. one `/project/<slug>` page per public project).
"Derived" helpers (`getStats`, `inferMainProject`, `resolveTechnologyIds`, …) just
compute from data already loaded — they aren't a data source.

| URL | File | Type | Data it consumes | Source |
|-----|------|------|------------------|--------|
| `/` | `pages/index.astro` | static | getPublicProjects, getSiteEntries, getMainProjects, getSeries, getTechnologies | A |
| `/list` | `pages/list.astro` | static | getMainProjects, getPublicProjects, getSiteEntries | A |
| `/map` | `pages/map.astro` | static | getMainProjects, getPublicJournalEntries, getPublicProjects, getSeries, getSiteEntries | A |
| `/log` | `pages/log/index.astro` | static | getPublicJournalEntries, getPublicProjects | A |
| `/log/<slug>` | `pages/log/[slug].astro` | dynamic — 1 per journal entry | getPublicJournalEntries, getPublicProjects | A |
| `/project/<slug>` | `pages/project/[slug].astro` | dynamic — 1 per public project | getPublicProjects | A |
| `/video/<id>` | `pages/video/[id].astro` | dynamic — 1 per video | getVideoIndex, getVideoBundle, getPublicProjects | D (+A) |
| `/research` | `pages/research.astro` | static | getCollection("research"), getPublicProjects | B (+A) |
| `/research/<slug>` | `pages/research/[slug].astro` | dynamic — 1 per research .md | getCollection("research"), getPublicProjects | B (+A) |
| `/garden` | `pages/garden/index.astro` | static | getGardenData, getPublicProjects | C (+A) |
| `/garden/<slug>` | `pages/garden/[slug].astro` | dynamic — 1 per garden note | getGardenData, getPublicProjects | C (+A) |
| `/admin` | `pages/admin/index.astro` | live editor (see note) | runtime Firestore via `src/scripts/admin.ts` | LIVE Firestore |
| `/admin/settings` | `pages/admin/settings.astro` | live editor (see note) | runtime Firestore | LIVE Firestore |
| `/404` | `pages/404.astro` | static | none | — |
| `/rss.xml` | `pages/rss.xml.ts` | endpoint (XML) | getCollection | B |
| `/llms.txt` | `pages/llms.txt.ts` | endpoint (text) | getCollection, getPublicProjects, getVideoIndex | A+B+D |
| `/robots.txt` | `pages/robots.txt.ts` | endpoint (text) | none | — |

> **The `/admin` exception:** the two admin pages are the *only* place a database is
> touched in the visitor's browser at runtime (a live Firestore editor, loaded via
> `src/scripts/admin.ts`). Every other page is fully static. Admin is also excluded
> from the sitemap (`astro.config.mjs`).

### Worked example — what happens for the homepage `/`

1. `src/pages/index.astro` owns `/`. Code between the `---` fences runs at build time.
2. At `index.astro:17-21` it calls `getPublicProjects()`, `getSiteEntries()`, etc.,
   imported from `src/lib/data.ts` (`index.astro:5`).
3. `data.ts` resolves those from **Source A** (snapshot → Firestore → backup; §3).
4. `index.astro` loops over the returned objects and writes HTML.
5. The HTML is wrapped by `layouts/BaseLayout.astro` (shell) + `components/Header.astro` (nav).
6. Astro saves the result to `dist/index.html`.
7. A visitor downloads that static HTML. No DB call in their browser.

## 5. The data-layer API (`src/lib/data.ts`)

This one file is the gateway between pages and stored data. The functions a page can call:

| Function | Returns | Reads from |
|----------|---------|-----------|
| `getPublicProjects()` | public projects, sorted | Source A |
| `getPublicJournalEntries()` | public log/journal entries | Source A |
| `getSiteEntries()` | combined projects+entries feed | Source A |
| `getMainProjects()` | the "product line" groupings | A + `lib/main-project-defaults.json` |
| `getSeries()` | series groupings | A + `lib/series-defaults.json` |
| `getTechnologies()` | tech/stack tags with counts | A + `lib/technology-defaults.json` |
| `getVideoIndex()` / `getVideoBundle(id)` | video list / one video | Source D |
| `getStats(projects)` | counts/totals (derived, pure) | — computes from its argument |
| `inferMainProject()`, `inferSeries()`, `resolveTechnologyIds()` | best-guess classification (derived, pure) | — |

If you want to understand the **business rules** (what "public" means, how a project
gets assigned to a product line, the default/guessing logic), this file plus the
existing `DOMAIN_PRIMER.md` are where to look.

## 6. The dead code (the "graveyard")

`node scripts/trace-reachability.mjs` reports **25 genuinely dead files** — an old
React single-page-app that was replaced by the Astro site but never deleted:

```
index.html  →  src/main.jsx  →  src/App.jsx
   →  src/components/*.jsx          (23 React components: Dashboard, PublicView,
                                     ProjectTable, VideoShowcase, JournalView, …)
   →  src/lib/auth.jsx
   →  src/lib/db.js  →  src/lib/firebase.js   (a runtime Firestore client)
   +  src/index.css
```

How to confirm it's dead, three independent ways:

- It never appears as a JS bundle in `dist/` after `npm run build`.
- No `.astro` file imports any `.jsx` component (`grep -rl "components/.*jsx" src/pages src/layouts`).
- `npx madge src/main.jsx` shows this cluster is reachable *only* from `main.jsx`,
  which itself is referenced only by the dead root `index.html`.

(Note: the `.jsx` components and `lib/db.js`/`firebase.js` are the dead Firestore
*client*. The live site uses the *snapshot file* via `data.ts` instead — see §3.
The exception is `src/scripts/admin.ts`, which is live and still uses Firestore.)

## 7. Glossary (terms used above)

- **Entry point** — where execution begins. Here: each file in `src/pages/`.
- **Import** — a line like `import X from "./y"`; means "this file uses that file."
- **Static site** — pages are pre-built into HTML files at build time; the server
  just hands out files. Opposite of a server that builds each page per request.
- **Build** — running `npm run build`; turns `src/` into the `dist/` folder.
- **Content collection** — Astro's built-in way to load a folder of Markdown files
  (here `src/content/research` and `src/content/garden`), defined in `content.config.ts`.
- **`getStaticPaths()`** — a function in a `[slug].astro` / `[id].astro` file that
  tells Astro "build one page for each of these items." That's how dynamic routes
  expand into many static pages.
- **Island** — a small piece of in-browser JavaScript on an otherwise static page
  (here only 3 exist: the garden graph, the homepage stack interaction, admin settings).
